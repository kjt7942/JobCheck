# Harness Engineering System - AGENTS.md (Map for Agents)

모든 에이전트가 이 리포지토리의 기준, 아키텍처, 규율을 이해하고 맥락을 신속하게 파악할 수 있도록 돕는 루트 레벨의 인덱스 파일입니다. 백과사전이 아닌 가장 중요한 '지도' 역할을 합니다.

## 1. 📖 지식 체계 위치 (Docs System)
현재 디렉토리 하위의 `docs/` 폴더에는 주요 의도와 맥락이 저장됩니다. 문서를 항상 최신화 상태로 유지(Doc Gardening)해야 합니다.
- **Design Docs (`docs/design-docs/`)**: 핵심 신념, 설계 원칙, 아키텍처 가이드 (엔지니어의 의도가 여기에 명시됨)
- **Execution Plans (`docs/exec-plans/`)**: 현재 진행 중인 작업 내역(`active`) 및 완료된 내역(`completed`) 관리
- **References (`docs/references/`)**: 외부 라이브러리나 툴체인 사용법 등 LLM이 참조할 최적화된 학습 레퍼런스

## 2. 🏛️ 아키텍처 제약사항 (Strict Architectural Constraints)
- **Layered Domain Architecture**: 비즈니스 도메인은 다음과 같이 고정된 계층으로 명확하게 분리됩니다.
  👉 **`Types -> Config -> Repo -> Service -> Runtime -> UI`**
- **의존성 규칙**: 각 계층은 자신보다 하위 컴포넌트(위 리스트의 좌측 방향)로만 의존 가능합니다. 역참조는 일절 불허합니다.
- **Providers**: 인증, 텔레메트리, 기능 플래그 등 애플리케이션 전체에 영향을 주는 공통 관심사(Cross-cutting Concerns)는 `Providers`라는 단일 인터페이스를 통해 주입(Injection)되어야 합니다.
- **제약 강제 적용 (Mechanical Enforcement)**: 위반 사항은 커스텀 린터 시스템 혹은 구조적 테스트에 의해 컴파일 또는 커밋/CI 단계에서 기계적으로 거부됩니다.

## 3. 🔍 에이전트 가시성 및 검증 (Observability & Validation Loop)
- 에이전트는 로컬의 각 PR 워크트리별로 앱을 독립 실행할 수 있어야 합니다.
- LogQL (로그 검증) / PromQL (메트릭 검증) 쿼리 등을 통해 자신이 짠 기능의 성능/안정성을 직접 검증합니다.
- 프론트엔드의 경우 Playwright와 스크린샷 캡쳐 등 Chrome DevTools 통합을 통해 시각적 버그 유무를 자율 재현, 검증할 수 있어야 합니다.

## 4. 🔄 자율 진화 루프 (Ralph Wiggum Loop)
- 변경 사항은 스스로가 작성하고 리뷰/테스트를 반복하는 자율 루프를 거쳐야 합니다. (완벽해질 때까지 스스로 피드백 사이클 적용)
- **Immutable Logic & Cumulative Updates**: 기존에 충분히 검증되고 작동 중인 핵심 로직을 함부로 훼손/제거하지 않습니다. 새로운 기능이나 최적화는 '기존 기능 위에 누적(Cumulative)' 하는 방식으로 업데이트합니다.
- **주의 (CAUTION)**: 엔지니어(대장님)의 직접적인 승인 없이 기존의 코어 구조나 안정적 토대를 근본적으로 뒤엎는 리팩토링을 단독 진행하지 마십시오.

## 5. ✅ 꼼꼼한 검증 및 완료 정의 (Definition of Done)
- **최종 시연 검증**: 모든 기능 구현 후에는 반드시 브라우저 서브에이전트나 로컬 실행을 통해 '사용자 시나리오'대로 작동하는지 최종 확인해야 합니다. 
- **문서화 정원 가꾸기 (Doc Gardening)**: 작업이 끝나면 `docs/exec-plans/completed/`에 작업 리포트를 작성하여 하네스 시스템의 지식 체계를 업데이트해야 합니다.
- **예외 처리의 철저함**: 데이터 저장 등 외부 API 연동 시 실패 케이스에 대한 방어 로직이 포함되었는지 다시 한번 점검하십시오.

## 6. ⚡ 토큰 효율성 및 테스트 제약 (Efficiency & Testing Constraints)
- **테스트 횟수 제한**: 단일 프로젝트 작업 시 브라우저 검증 및 자동화 테스트는 **최대 3회** 이내로 제한합니다.
- **심층 분석 우선**: 반복적인 테스트보다 코드와 로그를 통한 심층적인 원인 파악을 우선합니다. 한 번의 테스트로 발생 가능한 모든 문제점을 파악할 수 있도록 검증 시나리오를 촘촘하게 설계해야 합니다.
- **토큰 최적화**: 불필요한 파일 조회나 반복적인 툴 호출을 지양하고, 통합적인 해결책을 도출하여 토큰 소비를 최소화합니다.
- **검증의 질 향상**: 테스트 수행 전 가설을 명확히 하고, 테스트 후에는 로그와 스크린샷을 통해 다각도로 결과를 분석하여 재현 비용을 줄입니다.
